==================================================
1) CONTEXTE ET OBJECTIF
==================================================
Ce rapport presente la demarche de deploiement continue d une application web sur une infrastructure cloud.
L objectif principal est pedagogique: montrer comment automatiser la construction, la livraison et la mise en ligne avec un pipeline CI/CD fiable.

Ce document se concentre sur:
- la methode de deploiement,
- la robustesse du pipeline,
- la gestion des incidents,
- la validation finale du service.

==================================================
2) CADRE PRISM (APPROCHE UTILISEE)
==================================================
Dans ce projet, nous utilisons la logique PRISM:

P - Preparation
- Preparation de l environnement (cloud, secrets, depots, acces SSH)
- Preparation des fichiers d orchestration et de configuration

R - Realisation
- Construction automatique des images
- Publication vers un registre de conteneurs

I - Integration
- Provisionning de l infrastructure
- Connexion du pipeline avec la machine cible

S - Stabilisation
- Verification de disponibilite SSH
- Relances controlees, timeouts et fallback
- Verification interne de sante des services

M - Monitoring / Maintenance
- Diagnostics automatiques en cas d echec
- Ameliorations incrementales du workflow

==================================================
3) ARCHITECTURE DE DEPLOIEMENT (VUE GENERALE)
==================================================
Elements principaux:
- Pipeline GitHub Actions (jobs de build, infra, deploy)
- Instance cloud (EC2)
- Orchestration par Docker Compose
- Registre d images (GHCR)
- Verification de sante apres deploiement

Resultat attendu:
Une nouvelle version est automatiquement construite, publiee, deployee puis verifiee.

==================================================
4) DEROULEMENT TECHNIQUE DU PIPELINE
==================================================
Etape A - Build
- Construction des images backend/frontend
- Push des images avec tags (SHA + latest)
- Mecanisme de secours si Buildx echoue

Etape B - Infrastructure
- Initialisation Terraform
- Application de l etat cible
- Recuperation de l IP publique et de l instance cible

Etape C - Deploiement
- Verification de l etat machine et de SSH
- Connexion distante securisee
- Pull des images et lancement compose
- Verification de sante interne puis externe

Etape D - Post verification
- Publication URL finale
- Collecte de diagnostics si erreur

==================================================
5) INCIDENTS RENCONTRES ET CORRECTIONS
==================================================
Exemple de problemes pedagogiquement interessants:

1. Timeout SSH malgre machine active
- Cause probable: service SSH non stable apres boot
- Correction: boucle d attente de banner + retries + fallback infrastructure

2. Blocage apparent pendant status containers
- Cause probable: commande non bornee
- Correction: timeout explicite + commande de secours + logs heartbeat

3. Sante backend lente
- Cause probable: temps de demarrage applicatif
- Correction: boucle de health check interne amelioree + observabilite

4. Echec Buildx lie au reseau externe
- Cause probable: timeout vers Docker Hub
- Correction: chemin fallback de build/push classique

==================================================
6) VALIDATION ET CRITERES DE REUSSITE
==================================================
Le deploiement est considere valide si:
- les images sont construites et publiees,
- l instance cible est joignable,
- les conteneurs sont en etat Up/Healthy,
- l endpoint de health renvoie HTTP 200,
- l URL finale est accessible.

Indicateurs a relever:
- duree totale du pipeline,
- taux de succes sur N executions,
- temps moyen de remediation en cas d incident,
- nombre d interventions manuelles restantes.
